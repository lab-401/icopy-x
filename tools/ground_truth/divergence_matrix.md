The trace is captured POST-`translate_response`, so the normalized form is what appears in `PM3<` lines.

Evidence:
- `_normalize_wrbl_response` source (pm3_compat.py:1226-1232):
  ```python
  text = text.replace('Write ( ok )', 'isOk:01')
  text = text.replace('Write ( fail )', 'isOk:00')
  text = re.sub(r'\(\s*ok\s*\)', 'isOk:01', text)
  text = re.sub(r'\(\s*fail\s*\)', 'isOk:00', text)
  ```
- Trace `trace_iceman_full_audit_v5_20260414.txt:883/885` shows BOTH `isOk:01` (normalized) and `Write ( ok )` (raw — normalizer may have been bypassed for that particular response, likely because the response arrived interleaved with a prior command's output).
- Trace `trace_iceman_autocopy_mf1k_hexact_20260414.txt:146-168` shows 175× `isOk:00` + 53× `isOk:01`, i.e. 175 FAILURES normalized from `Write ( fail )` and 53 SUCCESSES from `Write ( ok )`. `ret=1` is PM3-responded flag, not write-success flag.

**Iceman source truth:** `Write ( ok )` = success; `Write ( fail )` = failure. No `isOk:` emission. The adapter rewrites on purpose to preserve legacy semantics.

**Challenger C1 verdict: AGREE.** The iceman-labelled `isOk:00/01` trace data is a NORMALIZED view, not raw firmware output. v1's "SEMANTIC inversion" claim is wrong — there is no inversion at the firmware layer. Middleware regex `r'isOk:01|Write ( ok )'` works because the alternation catches both the normalized form (post-adapter) AND the raw form (pre-adapter in edge cases).

## Systemic divergences

### 1. Hex case divergence (`%02X` iceman vs `%02x` legacy)

**Affected commands** (every command that emits hex UID/ATQA/SAK/data): `hf 14a info`, `hf 14a reader`, `hf mf wrbl`, `hf mf rdbl`, `hf mf rdsc`, `hf mf cgetblk`, `hf mfu info`, `hf mfu dump`, `hf iclass rdbl`, `hf iclass info`, `hf 15 info`, `lf t55xx detect`, `lf em 4x05 info`, `hf felica reader`

**Middleware impact:** All regex in middleware use `[A-Fa-f0-9]` — already case-agnostic. **None affected.**

**Action:** none needed.

### 2. Section-header formatting (iceman `---------- X ----------` vs legacy `------ X------------------`)

**Affected commands:** `hf 14a info`, `hf iclass info`, `hf mfu info`, `hf mfu dump`, `hf mf rdsc` (Sector trailer decoded separator), `hf felica reader` (FeliCa tag info)

**Middleware impact:** None — executor `_RE_SECTION_HEADER` strips ALL `[=] -----...` lines before middleware sees them. **None affected.**

**Action:** none needed.

---

### 3. Hint capitalisation (`Hint: Try` vs `Hint: try`)

**Affected commands:** `hf 14a info` (6 hints — L2754, L3353-3380), `lf search` (L1996).

**Middleware impact:** None — middleware does not parse `Hint:` lines.

**Action:** none needed.

---

### 4. `hf mf rdbl` — iceman delegates vs legacy inlines

**Affected commands:** `hf mf rdbl`, `hf mf rdsc`, `hf mf cgetblk`.

**Middleware impact:** regex `_RE_BLOCK_DATA_LINE` (data: line) matches iceman shape; `_RE_BLOCK_SPACED` matches legacy grid shape. Middleware tries BOTH via the `[A-Fa-f0-9]{32}` joined-block regex `_RE_BLOCK_DATA`.

**Action:** `_normalize_rdbl_response` adapter reshapes legacy grid → iceman `data:` — wired for all three commands in `_RESPONSE_NORMALIZERS`.

---

### 5. Help-text divergence

**Affected:** all commands under `-h`. **Middleware impact:** zero — iCopy-X middleware never invokes `-h` (help text isn't `PrintAndLogEx`).

**Action:** none needed. Catalogued for completeness.

---

### 6. Save-message case (`Saved N bytes...` iceman vs `saved N bytes...` legacy)

**Affected:** `hf mf csave`, `hf mfu dump`, `hf 15 dump`, `hf iclass dump`, `lf t55xx dump`, `lf em 4x05 dump`, `data save`.

**Middleware impact:** some middleware regex uses `saved N bytes` substring (lfem4x05.py:265 — `r'saved \d+ bytes to binary file\s*(.*)'`). **MATCHES LEGACY ONLY.**

**Action:** compat `_normalize_save_messages` lowercases `Saved` → `saved` on iceman output. Wired for the commands listed above but MISSING for `hf mf csave`.

---

### 7. Dotted-separator vs colon-separator for named fields

**Affected:** `Prng detection`, `Static nonce`, `Magic capabilities`, `Chip Type`, `Modulation`, `Block0`, `Password Set`, `Serialno`, `Chipset`, `Old UID`, `Xor div key`, etc.

**Middleware impact:** 15+ regex depend on `:\s+`. Legacy emits colon, iceman emits `.....` dotted leader of varying length.

**Action:** compat `_RE_DOTTED_SEPARATOR` normalizes the common case; per-field fixers (`_normalize_magic_capabilities`, `_normalize_t55xx_config`, `_normalize_em4x05_info`) handle the regex-consumed cases.

---

### 8. `isOk:NN` vs `Write ( ok )` / `Found valid key`

**Affected:** `hf mf wrbl`, `hf mf darkside`, `hf mf nested`.

**Middleware impact:** `r'isOk:01|Write \( ok \)'` alternation handles both. Darkside/nested key extraction via `r'Found valid key\s*[:\[]\s*([A-Fa-f0-9]{12})'` with `re.IGNORECASE` (hfmfkeys.py:275) works on both.

**Key finding (v2):** Iceman HEAD does NOT emit `isOk:` at all (grep confirmed zero matches in `/tmp/rrg-pm3/client/src/`). The `isOk:00/01` tokens that appear in iceman-labelled traces are produced by `_normalize_wrbl_response` (pm3_compat.py:1216) which rewrites iceman `Write ( ok )`/`Write ( fail )` to legacy-shaped `isOk:01`/`isOk:00`. No firmware-level semantic inversion; the adapter translation is deliberate.

**Action:** iceman-pattern middleware regex (alternation) + compat `_normalize_wrbl_response` + `_normalize_darkside_key` adapter.

---

### 9. Prefix stripping (`[+]`/`[=]`/`[!!]` etc.)

**Affected:** ALL commands running with TTY-attached legacy PM3.

**Middleware impact:** `executor._clean_pm3_output` strips them universally, so middleware sees prefixless lines always.

**Action:** none (handled at executor layer).
